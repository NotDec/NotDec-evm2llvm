# 变量清理

最终，EVM 中所有值都存储在 256 位字中。因此，当某个值的类型小于 256 位时，通常需要清理剩余比特。

Solidity 编译器会在所有可能受剩余垃圾比特影响的操作之前执行这类清理。例如，在把值写入 memory 之前，必须清掉剩余比特，因为 memory 内容可能会被拿去计算哈希，也可能作为消息调用的数据发送出去。同样，在把值存入 storage 之前，也必须清理剩余比特，否则脏值可能被观察到。

要注意，使用 inline assembly 访问不满 256 位的 Solidity 变量不算这类操作：如果你通过 inline assembly 访问这些变量，编译器不保证值已经被正确清理。

另外，如果紧接着的操作不会受到影响，我们也不会清理比特。例如，`JUMPI` 只把任何非零值视为 `true`，因此在把布尔值作为 `JUMPI` 条件之前，不需要先清理它。

除了上面的设计原则，Solidity 编译器在把输入数据加载到栈上时，也会清理这些输入。

下表说明了针对不同类型的清理规则，其中“higher bits”指的是当类型小于 256 位时剩余的高位比特。

| Type | Valid Values | Cleanup of Invalid Values |
| --- | --- | --- |
| `enum` of `n` members | `0` 到 `n - 1` | 抛出异常 |
| `bool` | `0` 或 `1` | 结果变成 `1` |
| signed integers | 高位应当是符号位 | 当前会静默做 signextend 变成合法值，也就是说高位全部设置为符号位；未来可能改为抛出异常 |
| unsigned integers | 高位应当为 `0` | 当前会静默把高位清零变成合法值；未来可能改为抛出异常 |

注意，合法与非法值取决于类型位宽。以 `uint8` 为例，这个 8 位无符号类型的合法值是：

```text
0000...0000 0000 0000
0000...0000 0000 0001
0000...0000 0000 0010
....
0000...0000 1111 1111
```

任何非法值都会把高位清零：

```text
0101...1101 0010 1010   非法值
0000...0000 0010 1010   清理后的值
```

对于 `int8`，这个 8 位有符号类型的合法值如下：

Negative:

```text
1111...1111 1111 1111
1111...1111 1111 1110
....
1111...1111 1000 0000
```

Positive:

```text
0000...0000 0000 0000
0000...0000 0000 0001
0000...0000 0000 0010
....
0000...0000 0111 1111
```

编译器会用 `signextend` 扩展符号位，负数时符号位为 1，正数时为 0，并覆盖高位：

Negative:

```text
0010...1010 1111 1111   非法值
1111...1111 1111 1111   清理后的值
```

Positive:

```text
1101...0101 0000 0100   非法值
0000...0000 0000 0100   清理后的值
```
